## 3. 后端模块详细设计

### 3.1 config.rs - 配置模块

**功能**：从 `.env` 读取配置

| 配置项 | 环境变量 | 用途 |
|--------|---------|------|
| database_url | DATABASE_URL | MySQL 连接串 |
| jwt_secret | JWT_SECRET | JWT 签名密钥 |
| server_host | SERVER_HOST | 监听地址，默认 127.0.0.1 |
| server_port | SERVER_PORT | 监听端口，默认 8080 |

**核心数据结构**：
```rust
pub struct Config {
    pub database_url: String,
    pub jwt_secret: String,
    pub server_host: String,
    pub server_port: u16,
}
```

**接口**：`Config::from_env() -> Result<Self, EnvVarError>`

---

### 3.2 middleware.rs - JWT 中间件

**功能**：签发与校验 JWT，作为 Actix-web `Transform` 挂到所有路由前。

**核心函数**：

| 函数 | 输入 | 输出 | 功能 |
|------|------|------|------|
| `issue_token(secret, user_id, username)` | secret: &str, user_id: i64, username: &str | Result<String, JWTError> | 签发 JWT，设置 7 天过期 |
| `parse_token(token, secret)` | token: &str, secret: &str | Result<Claims, JWTError> | 校验并解析 token，返回 Claims |

**核心数据结构**：
```rust
#[derive(Serialize, Deserialize)]
pub struct Claims {
    pub sub: i64,            // user id (subject)
    pub username: String,    // 用户名
    pub iat: usize,          // 签发时间 (Unix seconds)
    pub exp: usize,          // 过期时间
}
```

**中间件行为**：
- 请求抵达时，从 `Authorization` 头取出 `Bearer <token>`
- 验证失败或过期 → 返回 HTTP 401
- 验证成功 → 把 user_id 写入请求扩展（Extension），供 handler 读取
- `/api/auth/*` 和 `/health` 免校验（白名单）

---

### 3.3 response.rs - 统一响应与错误类型

**统一响应结构**：
```rust
pub struct ApiResponse<T> {
    pub code: i32,
    pub message: String,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub data: Option<T>,
}
```
- code=0 表示成功，非 0 表示失败
- message 为中文可读描述
- data 为业务数据

**错误枚举** `AppError`：

| 成员 | HTTP 状态 | 含义 |
|------|----------|------|
| BadRequest(msg) | 400 | 参数不合法 |
| Unauthorized(msg) | 401 | 未授权 / 登录失败 |
| Forbidden(msg) | 403 | 无权限 / 越权 |
| NotFound(msg) | 404 | 资源不存在 |
| Conflict(msg) | 409 | 冲突（如用户名重复） |
| InternalError(msg) | 500 | 服务内部错误 |

**辅助函数**：`get_user_id(&HttpRequest) -> Result<i64, AppError>`
- 从请求扩展中读取已解析的 user_id，失败返回 401

### 3.5 routes.rs - 接口处理器

每个接口的详细描述：

#### 3.5.1 健康检查：`GET /health`

**输入**：无
**输出**：HTTP 200 + `{ "status": "ok" }`

**用途**：供运维探测服务存活状态。

---

#### 3.5.2 注册：`POST /api/auth/register`

**请求体**：
```json
{ "username": "alice", "password": "12345678", "email": "alice@example.com" }
```

**处理步骤**：
1. 校验用户名长度 3-50，密码 ≥ 6，邮箱含 `@`
2. 调用 `get_user_by_username`，若存在则返回 `Conflict`
3. 使用 `bcrypt::hash(password, DEFAULT_COST)` 生成密码哈希
4. 调用 `create_user` 写入数据库
5. 调用 `issue_token` 生成 Token
6. 返回 `{ token, user: { id, username, email } }`

**成功响应**：code=0

---

#### 3.5.3 登录：`POST /api/auth/login`

**请求体**：`{ username, password }`

**处理步骤**：
1. 根据用户名查询用户
2. 用 `bcrypt::verify` 比对密码
3. 生成并返回 Token
4. 登录失败返回 401

---

#### 3.5.4 当前用户：`GET /api/auth/profile`

**输入**：JWT（通过中间件解析 user_id）
**输出**：`{ id, username, email }`

---

#### 3.5.5 获取分类：`GET /api/categories`

**输入**：无（JWT 隐含 user_id）
**输出**：分类数组 `{ id, name, type, is_system }`
**说明**：返回系统分类 + 用户自创建分类（按 name 排序）

---

#### 3.5.6 账单列表：`GET /api/transactions?month=2026-06&type=expense&page=1&page_size=20`

**输入**：`month(必填)`，`type(可选)`，`page(默认1)`，`page_size(默认20，上限100)`
**输出**：`{ items[], total, page, page_size, total_pages }`

**SQL 逻辑**：
```sql
SELECT t.*, c.name AS category_name
FROM transactions t JOIN categories c ON t.category_id = c.id
WHERE t.user_id = ? AND DATE_FORMAT(t.occurred_at, '%Y-%m') = ?
  AND (type_filter IS NULL OR t.transaction_type = type_filter)
ORDER BY t.occurred_at DESC
LIMIT (page-1)*page_size, page_size
```

---

#### 3.5.7 新增账单：`POST /api/transactions`

**请求体**：
```json
{
  "category_id": 1,
  "amount": 30.50,
  "type": "expense",
  "occurred_at": "2026-06-15 12:30:00",
  "remark": "午餐"
}
```

**校验**：
- amount > 0
- type 必须为 "income" 或 "expense"
- 分类必须存在，且 category_type 与 type 一致

---

#### 3.5.8 更新账单：`PUT /api/transactions/{id}`

**请求体**：`{ category_id?, amount?, remark? }`（三个字段都是可选覆盖）
**安全校验**：
1. 校验该账单是否存在
2. 校验账单 user_id == 当前 user_id
3. 校验新分类是否属于当前用户或系统分类

---

#### 3.5.9 删除账单：`DELETE /api/transactions/{id}`

**安全校验**：同更新，需先校验归属

---

#### 3.5.10 预算查询：`GET /api/budgets?month=2026-06`

**输入**：`month`
**输出**：预算列表，每条包含 `{ id, category_id?, category_name, amount, spent, remaining, usage_percent, month }`

**SQL 逻辑**：从 transactions 表汇总已用金额，并计算百分比（四舍五入取整）

---

#### 3.5.11 设置/更新预算：`POST /api/budgets`

**请求体**：`{ amount, month, category_id? }`
**说明**：category_id 为空表示总预算；同一 user_id + month + category_id 若已有记录，则更新原记录的 amount（upsert）

---

#### 3.5.12 删除预算：`DELETE /api/budgets/{id}`

**校验**：归属用户

---

#### 3.5.13 月度统计：`GET /api/statistics/monthly?month=2026-06`

**输出**：
```json
{
  "income": 8500,
  "expense": 1500,
  "balance": 7000,
  "month": "2026-06",
  "category_totals": { "餐饮": 800, "交通": 400, ... },
  "budget": {
    "amount": 3000,
    "spent": 1500,
    "remaining": 1500,
    "usage_percent": 50,
    "is_over": false
  }
}
```
> budget 字段在没有预算时为 null

---

#### 3.5.14 AI 月度报告：`POST /api/ai/report`

**输入**：`{ month }`
**输出**：`{ month, income, expense, balance, top_category, budget, suggestions[] }`
**建议生成逻辑**：
1. 若支出为 0：提示"本月暂无账单，可先录入收支"
2. 若存在最大支出分类：提示该分类金额
3. 若存在预算：按使用率给出不同级别的提醒（<80%可控；80-100%谨慎；>100%超支）
4. 若没有预算：建议设置预算
5. 若结余为正：建议储蓄或学习投入
6. 若结余为负：建议回顾调整

**兜底**：若所有条件都不触发，给出"保持良好记账习惯"

---

#### 3.5.15 智能分类：`POST /api/ai/classify`

**输入**：`{ remark, amount, type }`
**输出**：`{ category_id, category_name, confidence, source }`

**规则匹配**（支出型，关键词命中即返回；若均未命中则 fallback 到"其他"或同类型第一个）：
- 关键词"午餐/晚餐/早餐/外卖/奶茶/咖啡/食堂" → 餐饮
- 关键词"地铁/打车/滴滴/公交/高铁/机票/出租/加油" → 交通
- 关键词"淘宝/京东/购物/衣服/超市/拼多多" → 购物
- 关键词"书/课程/学费/资料/考试" → 学习
- 关键词"电影/游戏/会员/旅游/演唱会" → 娱乐
- 关键词"药/医院/门诊/体检" → 医疗
- 关键词"房租/水电/物业/宽带" → 住房

**置信度**：规则命中为 0.9，fallback 为 0.5

### 3.6 main.rs - 启动流程

```
1. dotenv() 加载 .env
2. env_logger::init_from_env 初始化日志
3. Config::from_env() 读取配置
4. MySqlPoolOptions::new().max_connections(10).connect(&database_url) 建立连接池
5. 构造 AppState
6. 装配中间件：Logger → Cors → JwtMiddleware
7. 注册路由：/health + /api/auth/* + /api/categories + /api/transactions*
                         + /api/budgets* + /api/statistics/* + /api/ai/*
8. bind (host, port) 并 run
```

## 6. 错误码设计

| code | 对应 HTTP 状态 | 典型场景 |
|------|---------------|----------|
| 0 | 200 | 成功 |
| 40001 | 400 | 请求参数非法（金额 ≤ 0、日期格式错误等） |

## 8. 关键注意点

1. **字段命名**：数据库中字段名为 `category_type`、`transaction_type`，前端 JSON 中可能简写为 `type`。
2. **跨域**：后端默认开启任意 Origin 的 CORS 允许，方便调试。
3. **Decimal 精度**：所有金额用 `Decimal`（rust_decimal 1.x）或 MySQL `DECIMAL(10,2)`，避免浮点误差。
4. **Token 过期**：前端接到 401 后自动清除 Token 并跳登录页。
